# Limitations and non-claims

## Current limitations

QSO-DIGITALIS does not currently provide:

- an approved product charter;
- accepted envelope, capability, topic, query, response, acknowledgment, revocation, provenance, or retention schemas;
- an accepted canonicalization or hashing profile;
- a tested validator or reference implementation;
- deterministic fixtures or compatibility suites;
- network transport, service discovery, persistent storage, or multi-user operation;
- credential, signing, approval, repository-write, payment, or deployment authority;
- accepted integrations with QSO-SEEKER, QuantumStateObjects, or QSO-FABRIC;
- a package, release artifact, SBOM, attestation, or deployment target;
- operational service-level commitments or support ownership.

## Interpretation rules

- The repository name does not establish a product identity.
- “Digital Consciousness Field” is an architectural candidate label, not a literal consciousness claim.
- Roadmaps and generated placeholder files are planning artifacts, not implemented capability.
- A successful documentation build proves only that the documentation renders under the selected toolchain.
- A schema build proves format conformance only; it does not prove security, privacy, semantic correctness, or downstream acceptance.
- A local reference path, if later approved, would not by itself authorize production deployment.
- Contract acceptance must never be presented as approval of a conclusion or external action.

## Unresolved decisions

- approve, revise, or retire the charter candidate;
- establish a unique responsibility relative to neighboring repositories;
- choose canonical terminology and supported users;
- approve data classification, privacy, retention, revocation, and deletion rules;
- approve license, support, review, release, and incident ownership;
- define canonicalization, hashing, identity, capability, and time semantics;
- define migration, compatibility, rollback, and retirement criteria;
- establish exact evidence required for downstream acceptance.

## Safe contribution boundary

Contributors may improve documentation, decision records, diagrams, threat models, fixture plans, and review checklists while the charter is unresolved. They must not:

- represent a proposal as approved;
- run scaffold materialization as evidence of implementation;
- create runtime, transport, storage, signing, scheduling, or deployment authority;
- widen neighboring repository responsibilities;
- introduce external mutation or production data;
- mark release gates complete without immutable evidence and human approval.
