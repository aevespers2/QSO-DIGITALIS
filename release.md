## Versioning

- No SemVer release begins before explicit charter approval.
- A documentation snapshot could later use `0.0.1-docs.1` only after publication approval, exact rendered accessibility review, and resulting-state verification.

- Mutable version aliases are prohibited for compatibility claims.
- Retirement documentation must not imply that a runtime ever existed.

## Rollback and withdrawal

Roll back a documentation candidate when it broadens scope, hides an unresolved vacancy or conflict, loses source provenance, introduces broken routes, lacks accessible alternatives, omits a material obstruction, or represents validation as approval. Restore the prior supported documentation state or mark the candidate withdrawn; do not restore stale implementation claims.

A future contract rollback must preserve historical identity while preventing unsupported consumers, revoked access, withdrawn claims, expired evidence, or incompatible generations from becoming current again.

A retirement or migration rollback must restore the prior approved documentation and repository state without restoring revoked authority, withdrawn releases, expired evidence, or historical scaffold claims. The restoration record must identify the exact prior head, failed transition, affected consumers, reviewer, and retained evidence.
