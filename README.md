# QSO-DIGITALIS

QSO-DIGITALIS is a **documentation-first charter candidate** for a bounded, content-addressed evidence-exchange contract within the QSO portfolio.

The repository does not currently provide a field runtime, transport service, shared-memory system, identity authority, credential service, storage platform, autonomous decision-maker, or production deployment. Its only active work is to determine whether a distinct repository role can be justified without overlapping QSO-SEEKER retrieval, temporal interpretation, Bridge transport, Repository `1` disposition, QuantumStateObjects runtime state, or QSO-FABRIC experiment evidence.

## Proposed bounded role

The charter question is whether QSO-DIGITALIS should document a profile for **sanitized, inert, versioned evidence references, interpretations, and purpose-limited projections** exchanged among portfolio components. Any accepted role must remain:

- content-addressed and hash-verifiable;
- non-executable and credential-free;
- capability-scoped and purpose-limited;
- explicit about identity, transformation, uncertainty, sensitivity, retention, correction, revocation, and provenance;
- subordinate to source, temporal, transport, disposition, runtime, experiment, and review authorities rather than replacing them;
- independently testable by exact version, digest, and pairwise/triple-overlap witnesses.

The [source-observation interpretation profile](docs/source-observation-interpretation-profile.md) separates source evidence, temporal assessment, interpretation, projection, transport, disposition, and consumption identities. The [obstruction and gluing register](docs/obstruction-and-gluing.md) records the unresolved interfaces, ownership vacancies, dependency breaks, and correction/rollback gaps that keep the route blocked.

## Explicit non-goals

QSO-DIGITALIS does not claim literal consciousness, sentience, awareness, independent personhood, unrestricted shared memory, implicit trust, raw network capture transport, autonomous approval, repository writes, payments, settlement, credential custody, or production infrastructure.

The repository is also bound by `QSO-CONSENT-CAPACITY-LOCK-v1`; documentation or future fixtures cannot be used to bypass explicit, informed, current, revocable consent or human review.

## Evidence states

Documentation uses the following states consistently:

- `PROPOSED` — described but not accepted;
- `REVIEW` — ready for bounded architectural review;
- `BLOCKED` — missing a required decision or evidence;
- `VERIFIED` — validated only for the exact stated source and scope;
- `WITHDRAWN` — no longer current;
- `HISTORICAL` — preserved as provenance, not current authority.

Passing documentation checks establishes only source consistency. It does not approve the charter, revive historical PR #2 or PR #5 as implementation candidates, authorize scaffold writes, approve Pages publication, release a package, activate a service, or grant operational authority.

## Accessibility and review

Evidence state, authority, retirement, rollback, record identity, and obstruction status must be communicated in text and must not depend on color, layout, motion, or a diagram alone. Source review is required now; keyboard, focus, 200%/400% zoom and reflow, contrast, reduced-motion, and assistive-technology review remain required against any exact rendered publication candidate.
